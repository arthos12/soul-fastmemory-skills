# fast-memory 测试方案（Test Plan）

## 目标
验证 fast-memory 是否真的做到：
- 结束/中断前能保存
- new session 后能恢复
- 分层存储准确
- 动态保存有效
- 不因记忆系统变得过重

## 测试 1：新会话恢复测试
### 目的
验证 `LAST_SESSION.md` / `SESSION_HANDOFF.md` / `MEMORY.md` 的恢复顺序是否有效。

### 步骤
1. 进行一轮真实工作
2. 形成明确主线、下一步、关键文件
3. 执行一次收尾或触发动态保存
4. 开新 session
5. 按 skill 规则恢复

### 通过标准
- 能恢复最近主线
- 能说出当前步骤、目标、下一步
- 不需要用户重新完整复述

## 测试 2：动态保存测试
### 目的
验证不等到“结束 session”也能增量保存。

### 步骤
1. 在同一主线内连续推进多个子点
2. 中途给出关键参数 / 关键文件 / 关键结论
3. 检查是否已增量刷新：
   - `SESSION_HANDOFF.md`
   - 必要时 `LAST_SESSION.md`

### 通过标准
- 关键变化被写下
- 即使突然 reset，也能恢复主线

## 测试 3：分类准确性测试
### 目的
验证不同内容是否进入正确存储层。

### 测试输入混合：
- 长期规则
- 近端主线
- 当前步骤
- 下一步 / blocker
- 当天细节
- 临时噪音

### 通过标准
- 长期规则 → `MEMORY.md`
- 近端主线 → `LAST_SESSION.md`
- 交接/下一步 → `SESSION_HANDOFF.md`
- 当天细节 → `memory/YYYY-MM-DD.md`
- 噪音不进入长期层

## 测试 4：丢失风险测试
### 目的
验证高信息量互动后是否还会“聊完就没”。

### 步骤
1. 进行一轮高信息量互动
2. 检查是否有结构化保存
3. 新 session 抽查恢复

### 通过标准
- 高价值信息未丢
- 主线恢复不依赖用户重讲

## 测试 5：轻量性测试
### 目的
验证 fast-memory 不会把日常任务拖慢。

### 步骤
1. 轻任务下只做最小必要保存
2. 中任务下做增量保存
3. 对比是否出现明显响应拖慢

### 通过标准
- 默认保存机制不应显著拖慢日常响应
- 若拖慢，说明动态保存粒度过重

## 结论标准
fast-memory 通过标准至少包括：
1. 能恢复
2. 能增量保存
3. 能正确分类
4. 不轻易丢失高价值信息
5. 不明显拖慢日常使用
