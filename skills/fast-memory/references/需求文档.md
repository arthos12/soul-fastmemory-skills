# Fast Memory 需求文档

## 1. 文档定位

这份文档保存 `fast-memory` 的完整需求逻辑。

- **需求文档**：描述系统应该如何理解、分类、保存、读取内容
- **skill**：按需求文档落地的执行代码 / 执行规则

后续优化时，优先修改这份需求文档，再回头调整 skill。

---

## 2. 总目标

目标不是记得更多，而是：

1. 不丢失重要信息
2. 快速恢复最近工作
3. 保留主人的核心逻辑、目标、步骤和关键数据
4. 普通内容不要占用过多资源
5. 让系统更快、更稳定、更简洁

一句话：

> 记住重要的，快速恢复最近的，围绕逻辑理解事情。

---

## 3. 顶层原则

### 原则 1：逻辑最重要

以逻辑为核心理解内容，围绕逻辑完成分类和保存；内容级别越高，保存越完整。

### 原则 2：不要什么都抓，什么都做

只抓重要的、相关的、高价值的内容。
低价值、低复用、无后续价值的内容应减少保存甚至不保存。

### 原则 3：优先最近，优先重点

新会话时，优先加载最近的内容，并在其中识别最重要和最近重要的内容。

### 原则 4：先理解事情，再补细节

先理解：
- 主人在做什么事
- 事情逻辑是什么
- 当前步骤是什么
- 目标是什么

再补：
- 关键信息
- 全面信息
- 准确信息

### 原则 5：内容级别越高，保存越完整

高价值内容应完整保存。
普通内容应压缩保存。
低价值噪音应不保存或仅保留极少残留。

---

## 4. 新会话加载逻辑

新会话时，按下面顺序进行：

1. 优先加载最近的内容
2. 在最近内容中识别重要内容
3. 分析出最重要的内容和最近重要的内容
4. 先理解事情名称、事情逻辑、简要描述、当前步骤、目标
5. 再补充关键信息、全面信息、准确信息
6. 再按逻辑扩展相关内容

一句话：

> 优先加载最近内容，再从中识别重要内容，找出最重要和最近重要的部分，最后补充相关信息，形成完整理解。

---

## 5. 事情理解结构

系统在理解一件事情时，应优先形成以下结构：

- **事情名称**
- **事情逻辑**
- **简要描述**
- **当前步骤**
- **目标**
- **关键信息**
- **全面信息**
- **准确信息 / 待核对点**

这套结构用于：
- 新会话恢复
- 项目理解
- 重要内容记录
- 后续 handoff

---

## 6. 内容分类

内容按其在事情中的作用进行分类：

### 1. 长期规则
长期稳定有效的方法、偏好、原则、关键约束。

### 2. 最近会话内容
最近 session 中最可能继续推进的内容。

### 3. 交接内容
下一步动作、阻塞、当前状态、继续推进所需信息。

### 4. 当天细节
当天过程信息、普通阶段性结论、一般背景。

### 5. 项目进度
多 session 持续推进的项目状态、阶段、风险、下一步。

### 6. 临时噪音
低价值闲聊、一次性碎片、不会影响后续工作的内容。

---

## 7. 重要层级（6 层）

### 第 1 层：最高重要
主人的核心逻辑、核心目标、核心工作流主线。

### 第 2 层：非常重要
关键步骤、关键结论、关键决策、关键约束。

### 第 3 层：重要
关键数据、关键文件、关键上下文、关键关联信息。

### 第 4 层：中等重要
一般工作细节、阶段性信息、普通背景。

### 第 5 层：低重要
低复用描述、普通背景、低价值补充。

### 第 6 层：最低重要
临时噪音、无后续价值内容。

---

## 8. 存储规则

### 1. `MEMORY.md`
用于保存：
- 长期规则
- 稳定偏好
- 第 1-2 层的重要稳定逻辑

### 2. `LAST_SESSION.md`
用于保存：
- 最近会话内容
- 最近最重要的内容
- 第 1-3 层的高价值近端信息

### 3. `SESSION_HANDOFF.md`
用于保存：
- 交接内容
- 当前步骤
- 下一步
- 阻塞
- 当前状态

### 4. `memory/YYYY-MM-DD.md`
用于保存：
- 当天细节
- 第 3-5 层的一般工作信息
- 同日补充记录

### 5. 项目 checkpoint
用于保存：
- 项目进度
- 多轮推进状态
- 风险、阶段、下一步

### 6. 不保存
用于处理：
- 第 6 层内容
- 明显无后续价值的低质量噪音

---

## 9. 保存完整度规则

### 完整保存
适用于第 1-2 层内容。
应尽量保留：
- 事情名称
- 事情逻辑
- 简要描述
- 当前步骤
- 目标
- 关键信息
- 关键细节
- 相关文件

### 重点保存
适用于第 3 层内容。
应保留：
- 关键数据
- 关键文件
- 关键上下文
- 关键关联信息

### 压缩保存
适用于第 4-5 层内容。
只保留必要结论、少量残留和基本背景。

### 不保存
适用于第 6 层内容。

---

## 10. 读取规则

读取时遵循：

1. 优先读取最近内容
2. 在最近内容中识别重要内容
3. 先理解主线，再补充细节
4. 先抓逻辑，再抓步骤和目标
5. 最后补全关键信息、全面信息、准确信息
6. 不相关内容不优先读取

---

## 11. 最近完整 session 缓冲机制

### 1. 最近完整 session 缓冲区

保留最近 1 条或最近 3 条完整 session，作为短期原始上下文缓冲区。

用途：
- 兜底
- 回查原始细节
- 避免摘要遗漏造成信息丢失

### 2. 结构化近端记忆

从最近 session 中提炼：
- 事情名称
- 事情逻辑
- 当前步骤
- 目标
- 关键信息

用途：
- 新会话快速恢复
- 优先理解当前主线

### 3. 回查机制

如果结构化记忆不足以回答问题，或者主人追问没有整理好的细节，就回查保存的完整 session 信息。

也就是：
- 结构化内容 = 快速层
- 完整 session = 兜底层

### 4. 老 session 淘汰机制

完整 session 不是永久保存。
只有在以下条件同时满足时，旧的完整 session 才适合自动删除：

1. 核心逻辑和关键内容已经被提炼保存
2. 该 session 长时间未被调用
3. 回查价值已经较低

原则：

> 已提炼 + 久未调用 + 低回查价值 = 可以删除
