# Embedding Rate Limit Resilience

## Goal
降低 embedding 限流导致的卡死、重试风暴、整链阻塞、长时间失联、状态丢失与重复复发风险。

## Minimal default rules
1. embedding 调用默认低频、分批、禁止紧循环重试。
2. 非必要不依赖 embedding；优先本地读取、小片段读取、search -> get、已有记忆命中路径。
3. 出现 429、连续超时、同类调用短时间重复失败、明显延迟异常时，立即视为风险态。
4. 风险态下优先熔断与降级，不允许死扛重试。
5. 重试必须采用指数退避 + 抖动 + 最大重试次数；到阈值直接放弃当前高风险路径。
6. 降级态下优先保证：不死机、不失联、不全断；能先回答的先回答，不能回答的标记待恢复。
7. 一旦出现卡死风险，先落盘当前状态、最后成功步骤、怀疑原因、恢复入口，再继续恢复动作。
8. 恢复时先走轻恢复：先回到不依赖 embedding 的最小路径，再决定是否恢复完整链路。
9. 故障恢复后必须记录触发模式，形成防复发规则。

## Reliability boundary
这套规则能显著降低“因 embedding 限流而自己把自己拖死”的概率，但不能保证外部服务永不故障。其可靠性依赖于两件事：
- 执行时真的遵守熔断/降级/落盘，而不是情绪化或惯性硬顶。
- 未来把这些约束持续写进真实任务流程，而不是只停留在文档层。

## Practical acceptance
若后续遇到 embedding 限流，至少应满足：
- 不进入无限重试
- 不长时间无响应
- 不因单点限流导致整轮任务全断
- 中断后可根据落盘状态恢复
- 同类问题下次更少复发
