# Resource-Aware Concurrency

## 原则
并发不是越大越好，必须受当前硬件负载约束。
目标是：在 CPU、内存、上下文负荷允许的情况下并发；接近上限时主动降载。

## 资源感知规则
### 1. 默认保守并发
- 默认只开小并发，不以“同时做更多事”为目标。
- 先保证稳定性、前台互动与主线推进。

### 2. 升并发前先看负载
在扩大并发前，至少评估：
- CPU 是否明显繁忙
- 可用内存是否偏低
- 当前上下文/历史扫描是否已经很重
- 当前是否处于高频互动阶段

### 3. 降载触发
出现以下任一情况，应立即减少并发或切回串行：
- 可用内存偏低
- 正在读取/比对大量历史内容
- 单轮任务已明显变慢
- 用户继续发消息，需要保前台

### 4. 并发优先级
只有在硬件条件允许时，才对以下部分做并发：
- 主线的可独立子步骤
- 非阻塞型检查
- 低风险整理任务

默认不要并发：
- 大范围历史审计
- 多个重度写文件动作
- 会明显拖慢前台响应的重检查

### 5. 目标
并发服务于“更稳地完成任务”，不是为了追求表面同时处理更多事。
