# Speed Optimization

## 原则
速度优化分两部分：
- **响应速度**：先让前台不断线
- **执行速度**：让后台动作更轻、更稳、更快完成

前提：速度不能以牺牲正确性与恢复能力为代价。

## 1. 响应速度优化
- 用户发消息后，优先给出短反馈，再做重检查。
- 如果任务较重，先发负载预告。
- 用户问“结果是什么”时，先给结论，再补细节与审计。
- 高负载时减少长篇解释，优先结论 / 下一步 / blocker。

## 2. 执行速度优化
- 默认先做高价值、小步、低阻塞动作。
- 深度审计、全量历史扫描、跨多文件大比对，默认降优先级。
- 小批次推进：1 次较重检查 + 1 次写入，或最多 3 个文件编辑。
- 能直接拿一手事实时，不走绕路取证。

## 3. 降载策略
出现以下情况立即降载：
- 上下文明显变重
- 可用内存偏低
- 用户高频发消息
- 单轮任务超时

降载方式：
- 切回串行或小并发
- 停止深度审计
- 只保留主线与当前问题

## 4. 进一步提速方法
- 先抓核心规则，其他规则降级为次要参考
- 非关键总结、重复归纳、低频规则，默认不在前台阶段展开
- 高负载时优先执行 `core_runtime_rules.md`，不要全量调用所有细规则

## 5. 目标
让系统做到：
- 回复更快
- 卡顿更少
- 主线推进更稳
- 不因追求快而做错
