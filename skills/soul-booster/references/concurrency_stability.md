# Concurrency Stability

## 当前问题
1. **前台被后台阻塞**：一旦进入大范围审计、长文档比对、多文件连续编辑，首条反馈会被拖慢。
2. **并发粒度过粗**：把“深度审计 + 文档改写 + 状态汇报”塞在同一轮里，容易卡住。
3. **缺少中间检查点**：长任务没有被切成可中断小步，导致用户一插话就像“断电”。
4. **结果导向不够前置**：先做很多动作，再想怎么汇报，导致互动停顿。
5. **高负载时不够主动降载**：任务重时仍试图同时做太多事。

## 改进方案
### 1. 三段式并发
每个重任务拆成三段：
- **A. 首条状态反馈**：先用一句话告诉主人正在做什么、为什么会慢。
- **B. 小步执行**：每次只做 1-3 个高价值动作，不连续堆很多操作。
- **C. 小结回报**：每完成一个批次，就立刻汇报“做了什么 / 结果怎样 / 下一步是什么”。

### 2. 小批次限制
默认每一轮后台推进只允许：
- 最多 3 个文件编辑，或
- 1 次较重检查 + 1 次写入
超过这个量，必须先回报再继续。

### 2.5 资源感知并发
并发不能无限放大，必须受 CPU / 内存 / 上下文负荷约束。
- 默认保守并发
- 负载升高时主动降载
- 只有硬件条件允许，才扩大并发
详见 [references/resource_aware_concurrency.md](references/resource_aware_concurrency.md)

### 3. 并发优先级
优先级固定为：
1. 当前用户互动
2. 主线任务下一步
3. 主线验证/测试
4. 支线文档整理
5. 深度审计/归档优化

### 4. 熔断规则
出现任一情况立刻停后台扩张并回报：
- 用户再次发消息
- 单轮任务超过预估时间
- 需要读大量历史消息
- 需要跨多个文件做大范围比对

### 5. 先结果后审计
如果用户问“结果是什么”，先给结果；
审计、追溯、完整证明放在后面补。

## 目标
让并发推进变成“前台不断线、后台不断做、小步快报”，而不是一次闷头做太多。
