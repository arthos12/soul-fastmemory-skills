# RQ0 需求采集与整理模型（Requirement Intake & Triage）

## 一句话定义
把碎片对话变成可执行需求清单：归类、去重、识别虚假/真实、判定重要度、明确产物与下一步。

## 输入
- 用户原话（可能碎片）
- 当前主线目标
- 现有 backlog（Single Source of Truth）

## 输出（强制字段）
对每条需求输出一个卡片：
- id/slug
- 原话（引用）
- 类型：P0大脑 / P1子能力 / P2业务 / 系统维护
- 真假：真实需求 / 虚假需求 / 不确定（需澄清）
- 重要度：S/A/B（S=影响主线与效率）
- 可执行产物：模型卡片/脚本/数据/验收
- 依赖：depends_on
- 状态：pending / mvc([m]) / done([x])

## 虚假需求识别（常见模式）
- 只是情绪宣泄/吐槽，没有可执行产物
- 只是泛泛愿望（"变聪明"）但未给验收/边界 → 需要转写成可执行指标
- 互相矛盾且未给优先级（需回到主次闸门）
- 只是一时好奇，不影响目标（降级到B或parking）

## 真实需求识别（强信号）
- 明确动作指令（"马上执行/修改/落地"）
- 明确验收（"我会检查占比/在规定时间内完成"）
- 明确约束/边界（"不要刷屏/不要依赖会话"）
- 明确收益目标（"提高效率/提高转化率"）

## 重要度判定（S/A/B）
- S：影响主线、效率、空闲推进、纠偏与真产出（例如：需求落地占比、自修复、主次闸门）
- A：直接产出业务能力或核心子能力（预测/监控/评分）
- B：解释性、一次性、低频，不影响主线

## 处理流程（最小闭环）
1) 采集→写入 backlog（否则视为没开始）
2) 去重/合并
3) 分型（类型/真假/重要度）
4) 产物化（MVC三件套）
5) 更新依赖关系与下一步指针
