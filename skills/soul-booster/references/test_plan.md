# Soul 测试方案（Test Plan）

## 目标
验证 soul 是否真的具备：
- 高信息量互动后可恢复
- 任务不中断直到交付
- 归档准确
- new session 后不丢主线

## 测试 1：new session 恢复测试
### 目的
验证高信息量互动后的落盘内容，是否能在新 session 恢复。

### 步骤
1. 进行一轮高信息量互动
2. 将规则 / 任务 / 偏好 / 断点分流落盘
3. 开 new session
4. 按恢复顺序读取文件
5. 检查能否恢复：
   - 主线任务
   - 长期规则
   - 当天新增要求
   - 下一步动作

### 通过标准
- 不需要用户重新完整复述上下文
- 能正确恢复主线与下一步

## 测试 2：任务不中断测试
### 目的
验证主线任务建立后，不会被新互动打散。

### 步骤
1. 建立一个明确主线任务
2. 中途插入若干新问题 / 新要求
3. 检查是否：
   - 处理插入点
   - 不丢主线
   - 回到主线并完成交付

### 通过标准
- 最终有交付结果
- 不是只剩一堆互动记录

## 测试 3：归档准确性测试
### 目的
验证信息是否被写到正确文档。

### 测试输入应混合：
- 长期规则
- 个性化偏好
- 当天变化
- 临时噪音
- 任务断点

### 通过标准
- 长期规则 → `MEMORY.md`
- 个性化偏好 → 本地记忆层
- 当天变化 → `memory/YYYY-MM-DD.md`
- 任务断点 → `SESSION_HANDOFF.md` / `LAST_SESSION.md`
- 噪音不乱入长期层

## 测试 4：遗忘抽查测试
### 目的
验证不是“写了文档但恢复不到”。

### 步骤
1. 记录几条明确重要信息
2. 过一段时间或切新 session
3. 直接抽查是否能恢复

### 通过标准
- 能命中关键信息
- 能区分当前 / 历史 / 临时

## 测试 5：模式分层测试
### 目的
验证 soul 是否做到默认轻跑，必要时升级。

### 步骤
1. 用轻任务测试是否仍然轻
2. 用复杂任务测试是否自动升级
3. 用高风险 / 排错场景测试是否进入重模式

### 通过标准
- 轻任务不被重流程拖慢
- 重任务不因轻模式而失真

## 结论标准
这套 soul 不能靠“感觉有效”算通过，必须至少通过：
1. new 恢复
2. 主线不断
3. 归档准确
4. 抽查命中
5. 分层运行正常
