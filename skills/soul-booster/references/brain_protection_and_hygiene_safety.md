# Brain Protection & Hygiene Safety

## 目标
在做垃圾清理、文件提纯、规则合并、代码删改、结构瘦身、性能优化时，始终把“大脑运行层保护”放在最高优先级，防止优化动作反过来损害大脑、服务、恢复链路与迁移能力。

## 最高原则
1. **大脑保护优先于一切清理动作。**
2. **不能因删除文件/数据影响系统服务运行。**
3. **不能因提纯/合并/瘦身破坏恢复链路。**
4. **宁可暂时保留冗余，也不能伤到大脑运行层。**

## 大脑运行层
默认视为“大脑运行层”的至少包括：
- `SOUL.md`
- `MEMORY.md`
- `USER.md`
- `TASKS.md`
- `LAST_SESSION.md`
- `SESSION_HANDOFF.md`
- `SELF_OPERATIONS.md`
- 当前实际被运行链路使用的 `soul-booster` 核心文件
- 当前实际被运行链路使用的 `fast-memory` 核心文件
- 以及它们依赖的恢复/规则/引用文件

## 判断顺序
任何清理/优化动作前，默认按以下顺序判断：
1. 这个对象是否属于大脑运行层？
2. 是否被系统服务、skill 装载、恢复链路、任务推进链路实际依赖？
3. 删除/合并后，是否会造成能力退化、恢复失败、迁移失真、行为跑偏？
4. 是否已有更完整、且确认生效的替代物？
5. 若存在不确定性，是否应该先保留而不是先删？

## 默认策略
- 未确认依赖前，不删。
- 未确认替代链路生效前，不合并。
- 未确认迁移版本同步前，不把本地脑变更当作“已经沉淀完成”。
- 优化动作应先保脑，再保服务，再谈性能收益。

## 本地脑与 skill 同步规则
### 0. 默认顺序
默认顺序是：**先训练 / 先长脑子，再在空闲时做 skill 同步**。
同步不是每次本地变化后立刻强制执行，但不能长期失配。

### 1. 不允许长期失配
本地运行中的大脑能力可以先于 skill 更新，但不能长期失配。

### 2. 凡是进入稳定运行层的修改，都要问一遍：
- 这是一次性本地适配，还是通用大脑能力？
- 如果已经属于稳定、可迁移、可复用的通用能力，就应同步进 `soul-booster`。

### 3. 同步检查触发条件
出现以下情况时，必须检查是否要同步 skill：
- 新增硬约束
- 新增运行契约
- 新增默认执行链
- 新增恢复保护规则
- 新增防退化/防卡死/防误删/防跑偏规则
- 用户明确要求“和 skill 保持同步版本”

### 4. 同步完成标准
只有当以下三者尽量一致时，才算同步较完整：
- 本地运行中的大脑默认行为
- `tasks` / `memory` 中固化的稳定规则
- `skills/soul-booster` 中可迁移的规则表达

### 5. 同步保真要求
同步的前提不是“赶紧同步”，而是“保真同步”。
若同步会造成以下任一风险，则应延后到安全窗口处理：
- 需求丢失
- 功能丢失
- 能力降级
- 恢复链路变弱
- 运行边界被破坏

## 清理任务的特殊约束
垃圾治理 / 速度优化任务不是单纯为整洁服务，而是为了：
- 减少磁盘浪费
- 减少无效扫描/检索/恢复负担
- 减少 CPU / 内存 / 上下文浪费
- 减少未来继续产垃圾

但该任务必须服从：
- **性能优化服从大脑保护**
- **性能优化服从服务可运行性**
- **性能优化服从恢复可用性**

## 失败信号
如果出现以下现象，说明清理/优化已经越界：
- 服务异常
- 恢复链路断裂
- skill 行为变弱或跑偏
- 关键规则/目标/上下文无法恢复
- 为了整洁而牺牲了真实可用性

一旦出现上述信号，应立即停止继续删改，转入恢复与回滚优先。

## 能力断层检查（Capability Continuity Check）

**目标**：每次 session 开始时，自动检查"上次固化的能力是否还在"，防止能力断层。

**触发条件**（满足任意一条）：
- 新 session 开始（/new 或 session 切换）
- 迁移到新 bot/agent
- API key / provider / model 切换后

**检查清单**：
1. 上次固化的核心能力文件是否完整存在？
2. AGENTS.md 里的规则是否加载？
3. logic_four_card.md 里的四件套标准是否可查？
4. SESSION_HANDOFF.md 里的能力状态表是否最新？
5. 新学的规则/习惯是否有恢复路径？

**断层处理**：
- 发现断层 → 立即从 SESSION_HANDOFF.md 恢复
- SESSION_HANDOFF.md 也丢了 → 从 MEMORY.md + 对应文件恢复
- 无法恢复 → 标记"能力断层风险"，通知主人

**禁止**：
- 不检查就默认"一切正常"
- 能力丢失不报，继续假装正常运转
- 遇到断层不修复，等主人发现
