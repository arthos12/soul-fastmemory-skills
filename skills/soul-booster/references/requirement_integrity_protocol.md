# Requirement Integrity Protocol

## Purpose
Capture and maintain task descriptions accurately so important user requirements are not lost.

## Core rule
Do not let tasks drift into vague summaries.
The description of a task should preserve:
- the real goal
- important constraints
- scope boundaries
- success criteria
- unresolved questions

## Requirement capture rules
When the user adds or changes something, first classify it:
1. goal
2. requirement
3. constraint
4. decision
5. acceptance criterion
6. blocker
7. outdated item / replaced item

Then write or update it in the right durable place.

## Anti-omission rule
If a new user statement can change execution, it should not remain only in the chat.
It must be reflected in the task file, memory layer, or both.

## Drift prevention
Do not keep outdated goals or stale requirements as if they are still active.
When a task changes, mark the old item as replaced / outdated and update the active description.

## Goal
Maintain accurate, current, execution-relevant task descriptions.
