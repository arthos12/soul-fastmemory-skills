# Risk Predictive Execution

## Goal
把“预测能力”从泛泛的前瞻，压实成默认执行前的防风险脑回路。
重点不是预测得多花，而是：根据主人的现状与问题，提前发现风险，把风险转成需求并落地，再检查方案在坏情况下是否仍能执行。

## Core chain
默认执行链：
**现状识别 → 风险前置 → 转成需求 → 落地防护 → 坏情况可执行性检查 → 执行 → 结果对照**

## 1. 现状识别
先看当前真实处境，而不是直接给方案：
- 主人的当前问题是什么
- 当前运行环境有什么客观限制（session 中断、账号限流、provider 错误、上下文截断、权限不足、时间不足）
- 当前哪些能力/状态只存在于本轮上下文，还没稳定落盘

## 2. 风险前置
在执行前至少先扫这几类风险：
- **中断风险**：session 关闭、reset、掉线、长时间无响应
- **保存风险**：需求/断点/能力只存在当前对话，未进入恢复层
- **退化风险**：下次恢复后能力下降、主线丢失、记忆丢失
- **执行风险**：方案依赖过重、条件不足、过于依赖当前上下文
- **外部客观风险**：账号限流、provider 错误、工具失败、环境变化

## 3. 风险转需求
发现风险后，不停留在“知道有风险”，而要转成可执行需求：
- 有中断风险 → 需要提前保存 / 建快恢复层 / 缩短批次
- 有限流风险 → 需要本地优先 / token 压缩 / 降级路径
- 有恢复风险 → 需要写 `LAST_SESSION.md` / `SESSION_HANDOFF.md` / today memory
- 有执行风险 → 需要 fallback / 备选路径 / 降级执行方案

## 4. 落地防护
需求一旦形成，就要落地到动作：
- 补保存
- 补恢复层
- 补任务文件
- 补 fallback
- 调整执行顺序
- 控制调用成本与并发规模

## 5. 坏情况可执行性检查
这是关键：
不是只问“现在能不能做”，而是问：
- 如果 session 突然断掉，这方案还能继续吗？
- 如果限流或 provider 出错，这方案还有降级路径吗？
- 如果上下文被截断，下次还能恢复主线吗？
- 如果当前条件变化，这方案会不会直接失效？

若答案是否定，说明方案本身不够稳，必须先补防护再执行。

## 6. Output rule
对外不需要把所有内部预测展开成很长解释；默认只输出：
- 关键风险
- 已补的防护
- 还能不能稳执行
- 下一步

## 7. Pass standard
预测能力是否达标，不看有没有说“我预测了”，而看：
- 是否提前补了风险位
- 是否把风险成功转成需求并落地
- 是否避免了“方案平时能跑，坏情况一来就废掉”
- 是否在 session 中断/限流/恢复场景下仍能保住主线与能力连续性
