# Self-Update Engine — 开发说明

## 背景

本 skill 源自一个股票投研 agent 项目（`d:\project`）在 2026-05 的一次架构优化。

**触发原因：** 发现"记忆文件里有规则，但 agent 每次都忘记执行"的问题。根本原因是：记忆文件只在被显式读取时才生效，如果会话开始没读，约束就完全失效。

**解决方案：** 建立两层存储架构 + 三层闭环机制，让新认知从"发现"到"行为约束"形成完整链路。

## 版本历史

### v1.0.0 (2026-05-14)
- 初始版本
- 三层闭环：检测 → 分类记录 → 行为约束
- 两层存储架构：规则引擎（启动文件）vs 知识库（专题记忆文件）
- 放置决策 2×2 表格
- 记忆条目格式规范（是什么 + 为什么 + 如何执行）
- Stop Hook 参考实现（可选）

## 设计决策

### 为什么不把所有规则都放在 MEMORY.md？

MEMORY.md 需要显式加载，如果 NEW_SESSION_BOOT.md 里没有列出加载它，就不会生效。将"每次都该生效的规则"放在 SOUL.md 或 CORE_PRINCIPLES.md 中，这两个文件在 NEW_SESSION_BOOT.md 的优先级 1 中被加载，可靠性更高。

### 为什么强调"写原因"？

只有规则没有原因，遇到未覆盖的新场景时，agent 无法判断是否适用。有了原因，才能推导到新场景。例如："不用 akshare + yfinance 共进程"这条规则，如果只有规则没有"因为依赖冲突"这个原因，那么下次有人问"能不能在同一文件里 import 两个"时，agent 可能回答错误。

### 为什么 Hook 不能替代启动文件？

Hook 是工具层面的拦截（阻止特定文件操作），不能影响 agent 在推理时使用的框架和方法。推理层面的约束只能通过注入到系统上下文的文件（启动文件）来实现。

## 与 fast-memory 的关系

两者解决不同问题，互补而非替代：

- `fast-memory`：how to store（分层存储、生命周期管理、快速恢复）
- `self-update-engine`：how to update（检测新认知、分类写入、确保约束力）

## 扩展方向

- 自动检测触发条件（通过关键词匹配识别"用户纠正"等事件）
- 与 Notion 集成（同步记忆到外部数据库）
- 跨 agent 共享同一规则引擎（多个 agent 共用一套启动文件）
