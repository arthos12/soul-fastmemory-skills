# Placement Guide — 记忆放置决策指南

## 核心问题

拿到一条新认知，问自己两个问题：

```
Q1: 这条规则每次会话都该生效吗？
    是 → 写进【规则引擎】（启动文件）
    否 → 问 Q2

Q2: 这是背景信息、推导过程、或历史记录吗？
    是 → 写进【知识库】（专题记忆文件）
    两者都是 → 启动文件写规则，记忆文件写原因
```

---

## 为什么要区分？

**规则引擎（启动文件）** — 每次会话自动注入，无需显式加载。
写在这里的规则，每次会话都会生效，即使 agent 没有主动去读记忆文件。

**知识库（专题记忆文件）** — 需要显式 Read 才能加载。
只在 agent 主动读取该文件时生效。如果会话开始没有读，内容不会影响行为。

**结论：** 只写在知识库里的规则 = 没有可靠的约束力。

---

## 放置示例

### 应该进规则引擎的内容

```
✅ 分析股票时必须先运行实时数据脚本，不能从记忆输出价格
✅ 代码文件路径全部用正斜杠
✅ API 调用必须包含超时和 try/except
✅ 声称完成前必须运行验证命令
✅ 不能在同一进程中同时 import akshare 和 yfinance
```

这些规则如果不进启动文件，每次会话都可能被遗忘。

### 应该进知识库的内容

```
✅ 为什么不用 akshare + yfinance 共进程（依赖冲突的具体原因）
✅ 某次踩坑的详细经过（方便理解规则背景）
✅ PEG 阈值选择的推导过程
✅ 某个 API 端点的具体行为特征
✅ 历史分析记录和方法论教训
```

这些内容太详细，放启动文件会导致文件膨胀、加载缓慢。

### 边界情况

**"我发现了一个编码规律，应该放哪里？"**

先问：这个规律每次写代码都需要应用吗？
- 是 → 进启动文件的编码规则模块
- 只是某个特定场景 → 进知识库的 feedback.md，附上场景说明

**"用户纠正了我的分析方法，放哪里？"**

两个地方都需要：
- 启动文件：加入"分析时禁止X，改用Y"这条规则
- 知识库：记录"用户说X不对的原因是..."

## 记忆条目格式

每条条目必须包含：

```markdown
- **规则**：[具体规则]
  **Why:** [原因 — 踩坑经历/用户明确说明/推导逻辑]
  **How to apply:** [在什么场景下适用，边界条件是什么]
```

**只写规则不写原因的条目** = 当遇到规则未覆盖的新场景时，agent 无法判断是否适用，会默认失效或错误应用。

---

## 精简 vs 准确的权衡

记忆文件应**准确性优先，简洁性其次**。

| 精简的代价 | 准确的好处 |
|-----------|-----------|
| 关键原则被删掉 | 边界情况能正确判断 |
| 规则失去背景，应用错误 | 下次遇到新场景时能推导 |
| 相似场景无法类比 | 规则的适用范围清晰 |

一条 5 行但包含原因的条目，远比 1 行但没有背景的条目更有价值。
