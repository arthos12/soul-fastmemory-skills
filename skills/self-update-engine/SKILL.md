# Self-Update Engine

## 定位

这个 skill 解决的核心问题是：**agent 学到的新方法/原则，如何真正改变未来的行为，而不是只停留在文件里？**

普通记忆系统只解决"记住"的问题，但不能保证"执行"。本 skill 通过三层闭环机制，把新认知从"发现"→"记录"→"约束行为"走通整个链路。

---

## 核心架构：三层闭环

```
发现新认知/原则
      ↓
第一层：检测（Detection）
  — 识别触发条件：用户纠正 / 踩坑 / 更好的方法 / 新框架
      ↓
第二层：分类记录（Classification & Recording）
  — 判断写到哪里（规则引擎 vs 知识库）
  — 写入时包含：是什么 + 为什么 + 如何执行
      ↓
第三层：行为约束（Behavioral Enforcement）
  — 规则引擎文件每次会话自动注入 → 立即约束
  — 知识库文件按任务类型显式加载 → 按需约束
```

---

## 第一层：检测 — 何时触发更新

以下任意一项发生，立即触发记忆更新（不等会话结束）：

| 触发事件 | 说明 |
|---------|------|
| 用户纠正 | "不对"、"不是这样"、"停止做X" |
| 用户确认非显然做法 | "对，就是这样"、接受了一个不寻常的选择 |
| 踩坑 | 代码报错、假设被推翻、API 行为与预期不符 |
| 发现更好模式 | 找到效率更高的写法/分析框架/工具用法 |
| 方法论更新 | 选股标准调整、分析维度扩展、排除逻辑变化 |
| SOP 发现缺口 | 某个步骤遗漏、流程有死角 |

**规则：发现即记录，不攒到最后。** 会话结束时的"统一总结"不如在每个认知出现时立即写入准确。

---

## 第二层：分类记录 — 写到哪里

### 两层存储架构

| 层级 | 名称 | 特点 | 约束力 |
|------|------|------|--------|
| 规则引擎 | 会话启动文件（SOUL.md / CORE_PRINCIPLES.md / NEW_SESSION_BOOT.md / CLAUDE.md） | 每次会话自动注入，无需显式加载 | **始终生效** |
| 知识库 | 专题记忆文件（MEMORY.md / memory/*.md / docs/*.md） | 需要显式 Read 才能加载 | **按需生效** |

### 放置判断（2×2 决策表）

| 问题 | 答案 | 写到哪里 |
|------|------|---------|
| 这条规则每次执行都该生效吗？ | 是 | **规则引擎（启动文件）** |
| 这是背景/历史/详细推导吗？ | 是 | 知识库（专题记忆文件） |
| 两者都需要？ | — | 启动文件写规则，记忆文件写原因 |

**关键洞察：只写在记忆文件里 ≠ 行为约束。** 记忆文件需要被显式读取才生效；如果会话开始时没有读到，约束就失效。重要规则必须进入启动文件才能可靠地约束每次会话。

### 写入格式要求

每条记忆条目必须包含三个要素：

```
① 是什么（具体规则或方法）
② 为什么（推导过程、踩坑经历、用户明确说明的原因）
③ 如何执行（在新场景下如何判断是否适用）
```

**只写"怎么做"不写"为什么"的条目，遇到边界情况时会被错误应用。**

### 内容分类映射（可根据项目定制）

| 内容类型 | 写入目标 |
|---------|---------|
| 每次都该执行的规则 | 启动文件（规则引擎） |
| 编码规范、踩坑记录 | feedback / 编码记忆 |
| 分析框架、方法论 | philosophy / 方法记忆 |
| 任务进度、阶段状态 | TASKS.md / 任务文件 |
| 决策历史、方法教训 | docs/lessons.md |
| 可验证的预测/判断 | docs/accuracy_log.md |

## 第三层：行为约束 — 确保规则真正生效

### 机制一：启动文件注入（核心）

在会话启动文件中（如 `NEW_SESSION_BOOT.md`）明确指定：不同任务类型开始前必须加载哪些文件。

示例结构：
```
股票分析类任务启动前：
  1. 加载投资哲学记忆（排除原则、估值框架）
  2. 加载方法论教训（避免重复错误）
  3. 查看近期 git log（了解最新变化）

写代码类任务启动前：
  加载编码规范记忆（禁止事项、约定、踩坑经历）

查看任务类：
  加载任务文件
```

这样，"按需约束"变得可预期——任务类型决定加载什么。

### 机制二：Stop Hook（可选，推荐）

在 agent 工具链支持 Hook 的环境中，配置 Stop Hook：

```
每次响应后 → 检测是否有文件改动（git diff）
→ 有改动 → 弹出记忆自检提醒（stderr，不消耗 token）
→ 无改动 → 静默，不干扰
```

**重要：hooks 只能约束工具行为（如阻止文件写入），不能约束推理层面的分析方法。** 推理层面的约束必须通过启动文件的内容来实现。

### 机制三：Verification Gate（声称完成前验证）

在声称任何工作"完成"之前，运行验证命令并读取输出。不能因为"代码看起来对了"就说完成。需要用真实数据或测试证明。

```
没有新鲜的验证证据 → 不能宣称完成
```

## 安装后的行为验收

安装本 skill 后，agent 应展现以下行为：

- [ ] 会话中发现新认知时，**立即**（而非会话结束时）更新对应记忆文件
- [ ] 新规则写入时，判断是进启动文件还是专题记忆文件
- [ ] 每条记忆条目包含：是什么 + 为什么 + 如何执行
- [ ] 不同任务类型开始前，加载对应的专题记忆
- [ ] 声称完成前运行验证，不说"应该可以"
- [ ] 记忆文件保持准确性优先，不因追求简洁而删掉关键原则

## 与 fast-memory skill 的关系

`fast-memory` 解决的是：**如何分层存储、快速恢复**（存什么、存哪里、怎么检索）。

`self-update-engine` 解决的是：**如何让记忆真正影响行为**（新认知如何从"发现"变成"约束"）。

两者互补，建议同时安装：
- `fast-memory` 处理存储和恢复
- `self-update-engine` 处理更新和约束

## 不存储的内容

以下内容不应进入记忆文件：

- 个股当前价格/PE/市值（运行时数据，从脚本查）
- 任务完成记录（进 git log，不进记忆）
- 本次会话的临时上下文（会话结束后无价值）
- 代码模式、架构设计（从代码库推导，不需要记忆）

## 常见反模式

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| 把重要规则只写在记忆文件 | 下次会话没读到，规则失效 | 重要规则进启动文件 |
| 只写规则不写原因 | 边界情况判断错误 | 每条规则附原因 |
| 过度精简记忆条目 | 关键细节丢失，下次重犯错误 | 准确性 > 简洁性 |
| 会话结束才统一写回 | 中途认知丢失，写回不准确 | 发现即写入 |
| 声称"应该可以"就算完成 | 交付有 bug 的结果 | 运行验证命令看输出 |
| 认为 hook 能约束推理行为 | hook 只能拦截工具调用，推理不受影响 | 推理约束靠启动文件内容 |
