# Foresight Persistence Audit

## Goal
检查“预判 / 预测能力”是否只是写在规则层里，还是已经进入可恢复、可执行、可验收的运行层；并补齐缺口。

## Current finding
当前 `preemptive_foresight.md` 已定义较完整：目标检查、路径识别、失败点、依赖、备选路径、确认边界、结果对照、迁移层判断。
但现阶段主要问题不是“完全没有”，而是：
- 预判能力更多存在于 rule/reference 层
- 快恢复层未把它列为当前专项检查对象
- 空闲推进清单中此前没有单独的“预判保存检查”动作
- 缺少明确的验收口径：如何判断预判能力真的恢复并在运行中被调用

## Required persistence layers
预判能力要想不易丢，至少同时存在于：
1. `SOUL.md` / `soul-booster`：作为底层能力定义
2. `TASKS.md`：作为待检查/待补强事项
3. `LAST_SESSION.md` / `SESSION_HANDOFF.md`：在当前阶段被明确列为专项检查项
4. `memory/YYYY-MM-DD.md`：记录本轮发现与修复动作
5. 需要时写入验收文档：说明如何检查其是否真实生效

## Audit checklist
- 是否能在当前主线中明确说出“预判能力也在当前待补强清单里”
- 是否在做方案时主动检查：失败点 / 依赖 / 备选路径 / 确认边界
- 是否把“预测命中/漏判”的结果在事后回流到文档
- 是否在新 session 恢复时还能把预判能力当成当前工作能力，而不只是历史概念

## Operational rule
以后凡是涉及重要方案、恢复链路、迁移设计、真实验收：
- 默认先做最小预判
- 不做空转式过度预测
- 预判必须落到：主路径、失败点、依赖、fallback、确认边界

## Pass standard
预判能力达标不以“写过文档”为准，而以以下行为为准：
- 做方案时会自然带出失败点与 fallback
- 快恢复层会记得它是当前专项能力之一
- 新 session 恢复后不会把它遗忘掉
