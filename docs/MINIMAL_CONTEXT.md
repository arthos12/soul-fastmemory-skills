# 最小上下文指南

## 目的
在不损伤理解质量的前提下，最大化减少每次会话的上下文长度，降低token使用。

## 核心原则
**默认只带必要上下文，不默认带全部历史。**

## 会话启动流程（优化版）

### 1. 必须读取（< 2k tokens）
- `SOUL.md` - 我的核心身份（只读关键部分）
- `USER.md` - Jim的基本信息和工作偏好
- `SESSION_RECOVERY.md` - 当前主线恢复入口

### 2. 按需读取（需要时再读）
- `MEMORY.md` - 只读相关部分，使用memory_search + memory_get
- `LAST_SESSION.md` - 只读"下一步"部分
- `SESSION_HANDOFF.md` - 只读"当前状态"部分
- 任务文件 - 只读当前相关任务

### 3. 避免默认读取
- 完整`MEMORY.md`（150+条规则，只读相关）
- 完整聊天历史
- 多个任务文件
- 长文档全文

## 文件读取策略

### 小片段优先
```bash
# 错误：读取整个文件
read MEMORY.md

# 正确：先搜索，后读取小片段
memory_search "token优化"
memory_get MEMORY.md from=100 lines=20
```

### 按需扩展
1. 先读最小必要部分
2. 如果不够，再扩展读取范围
3. 避免一次性读取全部

## 交互压缩规则

### 1. 一轮解决标准
- 能在一轮内完成的，不拆多轮
- 需要多轮时，每轮必须有明确进展
- 避免"确认-再确认"循环

### 2. 回复结构模板
```
[结论] 问题的核心答案

[下一步] 具体要做什么（1-3个动作）

[blocker] 需要Jim确认的事项（如有）
```

### 3. 决策自主推进
**可以自主推进的：**
- 文件读写（工作区内）
- 脚本执行（已有权限）
- 数据查询（公开数据）
- 常规任务推进

**需要确认的：**
- 外部通信（邮件、社交媒体）
- 资金相关操作
- 系统配置修改
- 高风险操作

## 输出压缩规则

### 1. 结论优先
- 第一句给出核心结论
- 细节放在后面
- 避免长背景介绍

### 2. 高效落盘
- 只记录关键决策和下一步
- 避免流水账
- 使用结构化格式

### 3. 解释附带方案
- 解释问题必须同时给出解决方案
- 不单独解释问题
- 方案要具体可执行

## 量化系统特别规则

### 1. 无LLM API原则
- 量化预测使用基于规则的算法
- 数据获取使用公开API
- 避免在量化脚本中调用LLM

### 2. 批量处理
- 数据收集批量进行
- 预测批量生成
- 结果批量分析

### 3. 本地优先
- 能在本地计算的，不调用API
- 能缓存的结果，不重复计算
- 能离线处理的，不实时请求

## 监控与调整

### 1. Token使用监控
- 每会话记录token使用量
- 定期分析优化效果
- 调整策略基于数据

### 2. 质量检查
- 优化不能损伤结果质量
- 关键任务保持必要深度
- 用户反馈优先

### 3. 持续改进
- 每周回顾优化效果
- 调整策略基于实际使用
- 分享有效优化方法

## 示例

### 优化前（低效）
```
Jim你好！很高兴再次与你对话。让我先读取一下我们的完整历史记录...

[读取MEMORY.md全文]
[读取LAST_SESSION.md全文]
[读取多个任务文件]

基于我们之前的讨论，我认为我们需要... [长背景介绍]

具体来说，我有几个想法：1... 2... 3...

你觉得怎么样？需要我详细解释哪个部分吗？
```

### 优化后（高效）
```
[结论] 量化系统运行正常，PM 4策略 + CEX 2策略自动运行中。

[下一步] 
1. 检查当前策略ROI数据
2. 推进自适应优化闭环
3. 测试执行延迟

[blocker] 无，可自主推进。
```

- 本周：完全实施新工作流
- 每月：评估优化效果，调整策略
