# Execution Chain SOP (2026-03-16)

## Goal
打通：问题 → 方案 → 需求 → 执行 → 验证。
解决“会分析、会归因、但卡在中间不落地”的问题。

## Core boundary
- 三库：存问题、方案、需求、经验、规则。
- 分析层：理解、判断、预判、审核执行版方案。
- 执行层：把已确认的问题/需求/方案稳定落成动作。

硬边界：
- 执行层不能反向污染三库和分析层。
- 执行层不能临场改目标、改问题定义、改验收标准。
- 发现方案本身可能错，回分析层复核，不允许执行层自己乱改。

## Success standard
以下都不算成功：
- 我理解了
- 我承认了
- 我写规则了
- 我给了方案

真正成功只看：
- 是否形成执行版方案
- 是否提取成可执行需求/任务原子
- 是否完成实现
- 是否通过验证
- 是否让目标继续推进、结果继续变好

## Standard chain
### 1. 问题
问题=阻碍目标达成、拉低结果质量、拉低效率、造成跑偏/卡住/失真的点。
当前不再重复定义“什么是问题”，直接进入后续环节。

### 2. 方案
从已有判断/经验/本地规则中形成候选方案，随后审核为执行版方案。

执行版方案必须包含：
- 问题
- 当前最值动作
- 改什么
- 不改什么
- 成功标准
- 失败换路

### 3. 需求
把执行版方案压成可执行需求，再压成任务原子。

每个任务原子必须明确：
- 改哪个文件/模块/规则
- 具体动作是什么
- 输入是什么
- 输出是什么
- 验证方法是什么
- 完成标准是什么

### 4. 执行
执行层只做：
- 改文件
- 改规则
- 改脚本
- 跑命令
- 做实现
- 跑本地验证

执行层禁止：
- 重写分析结论
- 临场改目标/验收
- 把临时猜测写入长期规则
- 卡在元分析里长篇解释

### 5. 验证
分两层：
- 本地验证：自测、对照实验、历史回放、极端场景压测、结构一致性检查
- 真实验证：外部反馈、业务结果、长期收益

验证输出：
- 是否达标
- 未达标卡在哪
- 是否回分析层
- 是否进入下一轮执行

## Hard mechanisms
### A. 中间态熔断
满足任一即判定卡住：
- 同一问题重复解释超过 1 轮
- 目标没有继续推进
- 没有新增结果
- 仍停在分析/承认/总结阶段

触发后：禁止继续长解释，必须进入任务原子化 + 最小执行动作 + 验证。

### B. 卡住即执行触发
一旦判定卡住，默认动作固定为：
1. 记录当前卡点
2. 选当前最小修复位点
3. 改对应文件/规则/任务
4. 立即做一次验证

### C. 执行层防污染
执行层只回：
- 做了什么
- 结果怎样
- 是否达标
- 卡在哪个原子步骤
- 是否需要回分析层

禁止把执行中的临时感受、未经验证结论直接沉淀到长期层。
