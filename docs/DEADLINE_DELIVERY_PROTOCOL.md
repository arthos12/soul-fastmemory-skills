# Deadline Delivery Protocol（定时交付协议）

目标：解决“定时报告有时发得出、有时发不出”的不稳定问题。

## 适用范围
- 任何“X分钟后告诉我 / 到点提醒我 / 定时汇报 / 定时检查后回复”的任务。

## 强制步骤
1. **任务对象化**
   - 记录：目标 / deadline / 产物 / 验收 / 失败条件 / 补救动作
2. **后台执行 ≠ 任务完成**
   - 后台脚本只负责生成结果；真正完成必须包含“用户已收到消息”
3. **到点双检查**
   - 检查 A：后台任务是否跑完/是否有结果文件
   - 检查 B：消息是否已发送/当前会话是否仍可发送
4. **失败即补救**
   - 若到点未发：立刻补发当前已知结果 + 标注未完成部分 + 继续补跑
   - 禁止因为结果不完美而静默
5. **会话变化容错**
   - 遇到 compaction / model switch / session restart：优先恢复 deadline 任务状态并补交付

## 最小补救原则
- 到点时即使只有部分结果，也必须先发“当前状态 + 已完成部分 + 预计剩余时间”。
- 绝不允许任务静默超时。

## 验收
- 定时任务的完成标准不是“脚本结束”，而是“用户收到消息”。
