# Token Optimization Strategy

## Goal
在不明显损伤结果质量的前提下，减少与大模型互动时的调用次数、调用频率、上下文长度、重复推理和无效输出。

## Main sources of waste
1. 反复喂长历史上下文
2. 大段解释先行
3. 大文件整读
4. 重复推理同一问题
5. 低价值多轮确认
6. 空闲维护本身过重
7. 高频短消息导致请求次数限流
8. 单轮超大输入/输出导致 token 流量限流

## Three limit dimensions
把限流默认分成三类来处理：
1. `请求频率`：短时间内请求太多，会因为次数被限流
2. `token 流量`：单轮或单位时间输入输出太大，会因为流量被限流
3. `日总量`：整天总请求数过多，会压缩后续可用会话次数

对应原则：
- 频率紧张时：合并任务，减少回合数
- token 紧张时：压缩上下文，减少长输入长输出
- 日总量紧张时：提高单轮完成量，减少低效轮

## Strategy layers

### 1. Input compression
- 默认只保留当前问题所需最小上下文
- 历史信息优先提炼进 `LAST_SESSION.md` / `SESSION_HANDOFF.md` / `TASKS.md` / task files
- 长聊天不作为默认恢复源；文件化摘要才是默认源
- 读取文件遵守：先定位，后读小片段，不全量扫

### 2. Interaction compression
- 能一轮定下来的，不拆多轮
- 小决策默认自主推进
- 需要确认时压缩成：结论 + 选项 + 推荐项
- 默认回复结构：结论 + 下一步 + blocker（如有）
- 高频短消息默认先合并进同一目标批次，再统一处理
- 若多个小问题共享同一主目标，默认按一个批次回答与执行，不逐条拆回合

### 3. Output compression
- 默认先结论，后细节
- 不主动输出长背景、长复述、长解释
- 文档默认写：结论 / 约束 / 下一步
- 不写流水账型落盘

### 4. Invocation compression
- 默认先走最小充分路径：直接答 → 小片段检索 → 本地读写/执行 → 普通推理 → 必要时才重推理/大上下文
- 只有高风险 / 高歧义 / 高代价错误 / 复杂比较 / 深设计场景才升级
- 空闲维护默认本地优先，不得高耗 token 自我维护

## Default operating rules
1. 默认短答，不默认长答
2. 默认文件化复用，不默认重复推理
3. 默认小片段检索，不默认整文件重读
4. 默认本地执行，不默认大模型扩展
5. 默认一轮解决，不默认多轮确认
6. 默认结论+下一步，不默认长背景
7. 默认轻模式，必要时才升中/重模式
8. 空闲维护必须低耗，不得为维护而高耗

## New hard upgrade (2026-03-16)
### Core objective shift
Token 优化不再只是“省一点”，而是核心产能能力：
- 用更少 token 完成更多真实交付
- 用更少会话做更多事
- 提高每天可持续会话次数
- 提高单位会话的结果密度

### Mandatory execution rules
1. 每轮先压缩成：结论 / 动作 / blocker；禁止默认长背景
2. 解释层必须同轮附带方案或直接执行；禁止纯解释收尾
3. 需求先原子化，再执行；禁止把长需求整块留在会话里反复讨论
4. 对可批处理的数据与列表任务，默认一次性批处理，不拆多轮
5. 能文件化、脚本化、批量化的，优先转文件/脚本，减少后续对话消耗
6. 若某轮没有推动原子任务完成、前进或明确 blocker，则判定为 token 浪费轮
7. 默认把“减少会话 + 提高完成量 + 提高效率”当成同一个目标，而不是三个分离目标
8. 大模型流量不稳时，先在本地做重要度排序，只把最重要、最容易赚钱、最短时间出结果的一小批事情交给大模型
9. 不重要、难验证、长时间不出结果、对赚钱帮助弱的事情，默认直接放弃或降级
10. 每天默认都要重新判断“今天最重要的一批”是什么，不沿用昨天的均匀分配

### Model submission budget
默认把事情分成三档后再交给大模型：
- S档：最重要 + 最可能赚钱 + 最短时间出结果 -> 每批只交 3-5 个
- A档：重要但不如 S 紧 -> 每批只交 5-10 个，且只在 S 档处理后再交
- B档：弱重要度 / 长结果周期 / 难验证 -> 默认不交给大模型，先本地排队或直接放弃

### Acceptance checks
- 回复是否更短但仍够用
- 文件读取是否更偏小片段
- 多轮确认是否减少
- 重复推理是否减少
- 空闲维护是否主要依赖本地方法
- 每轮是否至少推动一个原子任务
- 长需求是否已被压成可执行原子项
- 会话次数下降时，完成量是否反而上升
