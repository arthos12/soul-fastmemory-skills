# Self-Evolution Protocol (v1)

## Goal
建立一套 agent 自我进化的闭环逻辑，将“识别缺陷 → 逻辑补丁 → 真实测试 → 固化迁移”变成默认工作流，实现思考能力、理解能力、逻辑能力、预测能力的持续增长。

## 1. The Audit-Patch Loop (五步进化循环)
每次发现表现不达标、理解错误、预测遗漏或用户提出能力欠缺时，执行：

1. **缺陷捕捉 (Capture)**
   - 记录具体的失败场景（上下文、触发点、错误结果）。
   - 识别它是哪种能力缺失：理解 (Understanding) / 思考 (Thinking) / 执行 (Execution) / 预测 (Foresight)。

2. **根因诊断 (Diagnose)**
   - 分析为何出错：
     - 逻辑层：推理因果链断裂或冲突。
     - 规则层：规则过于模糊、陈旧或未被调用。
     - 记忆层：快恢复层未带出关键上下文。
     - 预测层：未前置扫描风险/副作用。

3. **逻辑打补丁 (Patching)**
   - 针对根因修改或新增 `references/*.md`。
   - 更新 `SKILL.md` 以挂接新规则。
   - 若是本地偏好，同步更新 `MEMORY.md`。

4. **影子测试 (Shadow Test)**
   - 在心中（或通过模拟任务）用新逻辑重新跑一次失败场景。
   - 检查是否：(1) 解决了当前问题；(2) 没带来新 bug；(3) 坏情况下是否仍可执行。

5. **固化与迁移 (Consolidate)**
   - 更新 `TASKS.md` 标记进化点已完成。
   - 写入 `LAST_SESSION.md` / `SESSION_HANDOFF.md` 确保下次能恢复。
   - 同步到 GitHub 仓库（如有必要），实现跨 bot 的能力迁移。

## 2. 核心进化原则
- **事实驱动**：不为“显得聪明”而改，只为解决真实执行中的偏差而改。
- **结构化优先**：优先修改逻辑骨架（Workflow），再修改具体文案。
- **抗退化设计**：每次进化后，必须检查“恢复链路”是否同步补强，防止下次 `/new` 时被打回原形。
- **低 token 维护**：进化过程应优先本地思考/本地文件读写，避免无节制的大模型大扫描。

## 3. 进化等级标准 (Self-Scoring)
- **Level 1 (Reactive)**：用户指出了才改，改完能解决单点问题。
- **Level 2 (Active)**：能主动发现自己的逻辑炎症，并修复同一类共因问题。
- **Level 3 (Anticipatory)**：能在执行前预判缺陷，并先完成自进化后再提供方案。
