# Task Classification & Change Control — 2026-03-17

## Why
碎片化、小改动、缺少分类会导致：
- 对“任务完整性”产生错觉（以为推进了，其实换了题）
- 本地落盘时把系统层改偏（把一次性需求写成硬规则）

目标：让执行更流畅，同时避免系统层漂移。

## Task types (必须先归类)
A. **Delivery / Output**：交付某个结果（数据/报告/脚本/下单/回填）。
B. **Research / Exploration**：探索性收集信息（允许发散，但输出要收敛）。
C. **Ops / Run**：持续运行闭环（抓取→预测→下单→回填）。
D. **System / Brain change**：修改底层规则、流程、记忆结构。

## Default handling by type
- A/B/C：默认直接执行到结果；只做最小落盘（数据/产物/指标）。
- D：必须走“变更闸门”，防止改偏。

## System change gate (for type D)
任何要写入 `SELF_OPERATIONS.md` / `MEMORY.md` 的改动，必须满足：
1. **清晰触发条件**：什么时候启用/不启用。
2. **可验收**：如何判断这条规则确实提升了结果。
3. **范围边界**：适用哪些任务类型；不适用哪些。
4. **最小化**：能放到 `docs/` 或 `tasks/` 的，不进 MEMORY。
5. **去重/不膨胀**：优先修改旧条目而不是新增一堆同义规则。

## Small updates policy
- 碎片需求默认：先归类→先做 A/B/C 的执行→再决定是否升级成 D。
- “系统层”不跟着碎片需求频繁抖动；除非出现**重复性故障**。

## External comms format (减少不明确)
对 Jim 的输出固定：
- **结果**（产物/数据/指标）
- **边界**（必须拍板/权限/资金）
- **下一步**（我马上做什么）
